## Independent Architecture Review Corrections (round 1 — how each finding is resolved)

The review accepted the purpose, reuse strategy, UI direction, capability isolation and combined-readiness objective, and named eight architecture corrections. Each is resolved below and carried into the design (§ references are to this revised plan).

- **F1 (P1) — labour conservation not globally enforceable.** The round-0 ledger keyed movements partly by `activityId`, so one crew could be allocated to two activities in the same date+shift. **Resolved (§C; refined round-2 finding 1):** the **`Worker` is the atomic capacity source** and capacity is conserved **at the worker**, globally, by `(projectId, workerId, civilDate, shift)` **before** any activity assignment; allocating a `Crew` EXPANDS transactionally into one `WorkerAllocation` per active member, so there is no crew-level allocation row. A partial-unique / exclusion constraint on `status='active'` rows makes a second live allocation of the same worker for the same date+shift **unrepresentable**; the competing-activity, worker-vs-crew and overlapping-crews races (required probes) prove exactly one allocation wins.
- **F2 (P1) — mixed units.** The round-0 model moved one `qty` through `committed→allocated→present→deployed`, conflating headcount and man-days. **Resolved (§B/§C):** readiness is measured in **person-shift / headcount**; actual work is a **separate** fact in **worked-minutes / man-days**. Attendance (a presence observation) and deployment (an effort observation) are **distinct immutable facts with distinct units — never transfers of the same quantity**.
- **F3 (P1) — requirement ownership + capability routing.** **Resolved (§B/§D):** **Activities** owns the requirement root/revision/events; **Labour** owns `LabourRequirementSpec`, written in the SAME transaction through an explicit `LabourParticipant` (the owner-aligned participant channel). The requirement command routes **by type**: `type='material'` asserts the `materials` capability and writes the material detail; `type='labour'` asserts the `labour` capability and writes the labour detail via the participant. The root `type` is immutable and PostgreSQL enforces **exactly one** correct detail record per revision.
- **F4 (P1) — revisions strand/reuse allocations.** **Resolved (§B/§C):** every commitment/allocation/satisfaction pins the exact `(projectId, requirementId, revision, labourSpecFingerprint)`; coverage counts only capacity matching the **current head** fingerprint (or an ACTIVE, current-fingerprint-bound, revocable skill substitution). A revision to an incompatible spec produces a new head fingerprint, so stale allocations **no longer satisfy** it (the activity re-derives to at-risk/blocked until re-allocated) — the requirement-revision-invalidation probe proves it.
- **F5 (P1) — trusted attendance identity.** **Resolved (§H):** a first-class project-contained `Worker` with active dates + revocation; `WorkerDevice` binds to a `Worker` by FK (free-text device name/trade is never readiness evidence); attendance is project-contained with evidence containment; a partial-unique gives **one attendance per `(worker, civilDate, shift)`** and the §C exclusion gives **no overlapping worker allocation**. Legacy aggregate `CrewRow` counts never influence readiness.
- **F6 (P1) — contradictory readiness table.** **Resolved (§A):** **two** deterministic first-match tables — **execution truth** (same-day, project timezone, injected clock; used by `activities.start`) requires allocation **and** presence; **forecast truth** (future need date; used by the projection/Inbox) requires allocation, with commitment→at-risk. When the labour capability is active, Team derives **entirely** from canonical coverage + unresolved mismatch facts; `Activity.gateTeam` is **legacy-only** (mutation rejected on capability-enabled projects), never a second writable truth.
- **F7 (P2) — duplicated supplier identity.** **Resolved (§F):** the round-0 `LabourSupplier` is **dropped**; the business party is the existing `Vendor`/`ProjectVendor`. Phase 4 adds only a labour-specific `VendorLabourProfile` (trades/skills supplied) and **separate** labour commercial documents (labour RFQ/PO — the commercial lines differ).
- **F8 (P2) — productivity has no numerator.** **Resolved (§I; refined round-2 finding 5):** the Daily-Log `progress`/photo counters are NOT measured output. Phase 4 defines **two owner-split facts** — Labour's `LabourWorkFact` (worked-minutes of effort) and Activities' generic `ActivityWorkOutput` (quantity + UOM + evidence); productivity is their derived join (measured output ÷ actual effort). Productivity **ships** in Task 5 because Phase 5 (cost + payment evaluation) depends on trustworthy measured work — it is no longer deferrable.
- **Mandatory review stop (instruction 12):** a **narrow review stop after Task 4** (the readiness core), **before** Daily-Log reconciliation and frontend integration.

- **Round-3 guardrail 1 — deadlock-free crew allocation.** `allocate` locks `Worker` rows `FOR UPDATE` in a **stable ascending `workerId` order** (§C.2/§H), so two overlapping crews sharing members acquire the shared worker locks in the same order and cannot deadlock.
- **Round-3 guardrail 2 — cumulative worked-minutes bound.** `work.record` enforces `Σ workedMinutes ≤ shiftMinutes` per `(worker, civilDate, shift)` by re-deriving the slice total under the worker `FOR UPDATE` (§C.4/§I) — **not merely a per-row CHECK**, which alone cannot stop multiple rows summing past the shift.

## Architecture (the corrected time-capacity design decisions)

### §A. Labour-readiness truth (execution vs forecast) + combined readiness + lock protocol

**Command authority is canonical, transactional and locked.** A new `LabourService.coverageFor(tx, projectId, requirements, asOf)` returns a per-labour-requirement verdict, read **in-tx under `lockProjectReadiness`** — the SAME lock material coverage takes, so labour and material writes on one project serialise on one advisory lock and `start` sees a consistent five-gate snapshot. `asOf` is the project-timezone civil date from an **injected clock** (never `new Date()`), so execution truth is deterministic and testable. `deriveTeamReading` maps the verdict to the gate (sibling of `deriveMaterialReading`). The labour-readiness projection (§G) feeds UI/Inbox/Dashboard/forecast ONLY.

**Team-gate replacement + legacy isolation (F6/instruction 9).** When the `labour` capability is **active**, `deriveReadiness`'s Team gate derives **entirely** from canonical labour coverage + unresolved labour-mismatch facts (§E); `Activity.gateTeam` is **not read** and **its mutation is rejected** (no second writable truth). When the capability is **disabled** (legacy projects), `gateTeam` remains exactly the stored stub it is today — byte-identical. A `GateOverride{gate:'team'}` (authority + reason + evidence + expiry) still supersedes, unchanged.

**Readiness is measured in person-shift / headcount** (integer count of workers of the required trade/skill for a `(civilDate, shift)`), never man-days. Actual effort (§I) is a separate unit.

**Execution truth — the deterministic first-match table `activities.start` uses (asOf = project-tz civil today; per activity; worst-wins across the activity's demand slices due today):**

| # (first match) | Condition (`asOf` = project-timezone civil today; per activity, worst-wins) | `gt` |
|---|---|---|
| 1 | an unresolved labour-mismatch fact exists for the activity | `fail` |
| 2 | the activity has no `type='labour'` requirement | `na` |
| 3 | **`asOf` precedes the requirement's earliest demand slice** — the labour window has not begun | `wait` (start refused — not yet due) |
| 4 | **any required slice dated `< asOf` is unfulfilled** (its allocated-and-present person-shifts < the slice's `personShiftQty` and it was not worked) — an overdue window | `fail` |
| 5 | for **every required `(civilDate = asOf, shift)` slice due today**: **allocated AND present** ≥ the slice's `personShiftQty` (head fingerprint or active substitution) — and every past slice was fulfilled | `ok` |
| 6 | today's slice(s) are **allocated** ≥ required but at least one required worker is not yet **present** (muster pending) | `wait` |
| 7 | otherwise (a slice due today is under-allocated) | `fail` |

Rows 3–4 close the round-0 **vacuous-ready path** (finding 3): a requirement whose window has not begun yields `wait` (start refused), and an overdue-unfulfilled window yields `fail` — so an empty "due today" set can never be vacuously `ok`. (A window fully in the past whose every slice was worked is genuinely satisfied — rows 3–4 do not fire and the worked-counts-as-coverage guardrail applies.)

**Forecast truth — the deterministic first-match table the projection/Inbox/Dashboard uses (for a future need date; per activity/slice; worst-wins):**

| # (first match) | Condition | forecast verdict |
|---|---|---|
| 1 | an unresolved labour-mismatch fact exists | `blocked` |
| 2 | no labour requirement | `not-required` |
| 3 | **allocated** capacity (head fingerprint / active substitution) ≥ required for the slice | `ready` |
| 4 | shortfall covered by a `CapacityCommitment` of the fingerprint dated ≤ `requiredBy` | `at-risk` |
| 5 | otherwise | `blocked` |

The two tables differ **only** in whether presence is required (execution) or allocation suffices (forecast) — resolving F6. Actual work already performed against the activity counts as satisfied capacity for that activity (the labour analogue of "issued material counts as coverage"; deploying an allocated crew never un-readies the activity).

**Combined material-and-labour activity readiness (accepted worst-gate composition).** Material and Team are independent derived gates; the activity is ready iff all five gates are `ok` (`readinessReady`). Material × Team subset (other gates `ok`):

| Material | Team | `start` | Surfaced |
|---|---|---|---|
| `ok` | `ok` | **allowed** | ready |
| `ok` | `wait` | refused | material available, **labour crew allocated but not mustered** |
| `ok` | `fail` | refused | **material available, labour BLOCKED** (under-allocated / no commitment) |
| `wait`/`fail` | `ok` | refused | labour available, material at-risk/blocked |
| `na` | `ok` | allowed¹ | no material demand |
| `ok` | `na` | allowed¹ | no labour demand |

¹ subject to the other three gates; a per-gate `pmc` override supersedes its row.

**Lock-coverage extension (the §A tripwire).** Every Phase-4 command that can move the Team gate takes `lockProjectReadiness` and joins `SECTION_A_COMMANDS`/`COVERAGE` in `readiness-lock-coverage.test.ts` (bumping `toHaveLength(22)` to the new count): labour-requirement `{create,revise,cancel}` (via the routed requirement command); `capacity.commit/revise/default`; `allocation.allocate/release`; `attendance.record/revoke`; `work.record` (does not move the gate but is lock-consistent); `labour_mismatch.resolve`. Concurrency probes (live PG, both orderings under the lock): **competing-activity crew allocation** (two activities, same worker/date/shift → exactly one wins); **worker-vs-crew** (the same worker allocated directly and via a crew → exactly one wins); **overlapping-crews** (two crews sharing a worker, both allocated for the same date/shift → exactly one wins); allocation-release vs `start`; attendance-record vs `start`; labour-requirement revision vs `start`; commitment-default vs `start`; plus the cross-gate probe (a material reservation and a labour allocation on one project serialise; `start` sees a consistent snapshot).

### §C. Time-bounded capacity conservation (NOT a stock ledger — F1/F2/instructions 1–3)

Labour is expiring capacity, so the model is **three immutable append-only fact families (commitment, attendance, work) + one frozen-identity CAS allocation lifecycle — each with a distinct unit** — not one mixed-unit quantity moved through buckets. **The atomic execution-capacity source is the `Worker` (finding 1); a `Crew` is organizational only.**

1. **`CapacityCommitment`** (inbound, forecast — aggregate) — a source (a `Vendor` labour supplier via §F, or an in-house pool) commits `personShiftQty` of a `labourSpecFingerprint` **per `(civilDate, shift)` slice** (finding 2 — never a bare "headcount over a range"; a range commitment is a set of dated slices, each stating its own quantity). Unit: person-shift. Aggregate forecast capacity that does not name a `Worker`; it becomes execution capacity only once named `Worker`s are onboarded and allocated. Immutable; a revision appends a new dated promise (append-only, §F).
2. **`WorkerAllocation`** (assignment; the conserved fact — a frozen-identity CAS row, **finding 4**) — assigns exactly ONE **`Worker`** (the atomic source) for exactly ONE `(civilDate, shift)` slice to exactly ONE `(activityId, requirementId, originRevision, labourSpecFingerprint)`. Unit: 1 person-shift. **Identity is frozen** (a trigger freezes `workerId, civilDate, shift, activityId, requirementId, originRevision, labourSpecFingerprint`); only `status` (`active → released`) and the release attribution mutate — the `PurchaseOrderVersion` frozen-identity CAS pattern, **not** append-only (so the plan no longer claims both). **Global conservation is enforced at the `Worker` source (finding 1/instruction 2):** a partial-unique / `EXCLUDE` on `(projectId, workerId, civilDate, shift) WHERE status='active'` makes a second live allocation of one worker for one date+shift **unrepresentable in PostgreSQL** — a worker allocated directly cannot also be double-booked. **Allocating a `Crew` is a convenience that EXPANDS transactionally into one `WorkerAllocation` per active member** in the same transaction (instruction 1); there is no crew-level allocation row, so a worker in two crews, or allocated directly and via a crew, still hits the one per-worker exclusion (the worker-vs-crew and overlapping-crews probes prove it). `allocate` takes `lockProjectReadiness` + `SELECT … FOR UPDATE` on each `Worker` **acquired in a stable ascending `workerId` order (round-3 guardrail — so two overlapping crews sharing members take the shared worker locks in the SAME order and cannot deadlock)**, checks the worker is active for the date (§H) and the fingerprint matches the requirement head, then INSERTs each member row; the loser of a race gets a deterministic 409/constraint violation. `release` CASes `active → released` (frozen identity retained) and frees the worker for re-allocation.
3. **`Attendance`** (presence observation — a DISTINCT fact, not a transfer) — worker `W` present on a `(projectId, civilDate, shift)` slice, with a `Worker` FK + `WorkerDevice` binding + evidence (§H). Unit: headcount (1 per worker). A partial-unique on `(projectId, workerId, civilDate, shift)` gives **one attendance per worker/date/shift** (finding 5). Attendance does **not** consume or move an allocation; coverage joins attendance to allocation at read time.
4. **`LabourWorkFact`** (effort observation — a DISTINCT fact, distinct unit; **Labour-owned**, finding 5) — worked-minutes of effort linked to `(projectId, workerId, allocationId, activityId, civilDate, shift)`, with `workedMinutes` bounded by the shift length **both per row (a CHECK) AND cumulatively (round-3 guardrail): the command re-derives `Σ workedMinutes` for the `(projectId, workerId, civilDate, shift)` slice under the worker `FOR UPDATE` and REFUSES any record that would push the slice total over `shiftMinutes`** — a per-row CHECK alone cannot stop multiple rows summing past the shift. Unit: worked-minutes / man-days. Effort is an **observation**, never a transfer of the presence headcount (F2). The generic measured construction output (`ActivityWorkOutput {quantity, uom, evidence}`) is a **separate Activities-owned** fact (§I), not this row.

**Compatible-revision carry-forward (finding 4).** A `WorkerAllocation` pins its `originRevision` for **provenance**; coverage (§A) carries it forward as satisfying the **current head** ONLY when the current head `labourSpecFingerprint` AND the allocation's `civilDate` AND `shift` all still match. Consequences (three required probes): a **responsible-only** revision (responsible is not in the fingerprint) keeps the same head fingerprint → the allocation still satisfies (preserved); a **trade/skill/shift/window** revision changes the head fingerprint (or the slice) → the allocation no longer matches head and is **stranded** (the activity re-derives to at-risk/blocked until re-allocated; the stale `active` allocation still holds the worker's capacity until explicitly released); a **headcount increase** (same fingerprint, larger `personShiftQty`) **preserves** the existing allocations as coverage but creates a **shortfall** for the added quantity.

There is **no current-quantity column and no bucket ledger.** Coverage (§A) is derived by *reading* these facts: allocated = `WorkerAllocation` rows with `status='active'` matching the head fingerprint + slice; present = allocated whose worker has an `Attendance` for the slice; committed = `CapacityCommitment` slices of the fingerprint dated ≤ requiredBy. Immutability, frozen-identity, and no-double constraints are DB-enforced (triggers, partial-uniques, `EXCLUDE`), never convention.

### §H. Trusted workforce identity + authorization + containment + concurrency (F5/instruction 7)

- **`Worker` (new, first-class, project-contained)** — `{ projectId, id, name, tradeCode, skillCodes[], activeFrom, activeTo?, revokedAt?, revokedById? }`; **the atomic execution-capacity source (finding 1)**. **`Crew`/`Gang`** — a named set of workers under an in-charge (`mistri`); **organizational only, NOT an atomic capacity source** — allocating a crew EXPANDS transactionally into one `WorkerAllocation` per active member (§C.2), so all capacity conservation is enforced per `Worker`. **`WorkerDevice` binds to a `Worker` by FK** (F5): the field attendance token carries a `workerId` FK; free-text device `name`/`trade` are display-only and **never** readiness evidence. ### §I. Planned vs actual + productivity (F8/instruction 11 + round-2 finding 5)

- **Two distinct facts, two owners (finding 5).** Effort and output are split by owner and never conflated into one row (F2). **Labour owns `LabourWorkFact`** — worked-minutes of effort linked to `(projectId, workerId, allocationId, activityId, civilDate, shift)`, `workedMinutes` CHECK-bounded by the shift (§C.4). **Activities owns the generic `ActivityWorkOutput { quantity, uom, evidence }`** measured-output fact (immutable, evidence-bearing) — the same generic construction-output fact any activity produces, NOT the Daily-Log photo/progress counter (F8) and NOT a labour-specific row. Neither is a transfer of the other's unit: effort is worked-minutes, output is a measured quantity in its own UOM.
- **Planned** = the requirement's person-shift demand by activity/date/shift (the explicit §B slices, not a bare `requiredHeadcount`). **Actual attendance** = the `Attendance` facts (headcount). **Actual effort** = the Labour-owned `LabourWorkFact` worked-minutes. **Measured output** = the Activities-owned `ActivityWorkOutput` quantity + UOM. Variance (attendance vs planned, effort vs planned) feeds the forecast projection.
- **Productivity is their derived join, composed by Activities, and it ships (finding 5 + round-3 acyclicity).** **Productivity** = measured output (`ActivityWorkOutput`) ÷ actual effort (`LabourWorkFact`) — a DERIVED read across their common `(projectId, activityId, civilDate, shift)`, never a stored column. **The composition belongs to the Activities side (the `ActivityWorkOutput` owner) — or a cross-cutting reporting projection — NEVER Labour (round-3):** Activities owns `ActivityWorkOutput`; Labour exposes effort through the query contract `LabourQuery.effortFor(tx, projectId, scope)`; the composing owner reads Labour's effort via that contract (the `activities → labour` read edge) and joins it with its own output. **Labour never reads `ActivityWorkOutput`** — that would be a `labour → activities` read edge and would close a cycle. Phase 4 **ships** this minimal two-fact productivity model (Task 5), because Phase 5 (cost + payment evaluation) depends on trustworthy measured work; it is no longer deferred. The join is a read only — no new writable truth and no new event beyond `labour_work.recorded` (Labour) and `activity_output.recorded` (Activities).
