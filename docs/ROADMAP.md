# Vitan PMC — Roadmap

Phased delivery, building forward from the [architecture](./ARCHITECTURE.md). Status reflects the current branch.

## Internal named-user password access (IMPLEMENTED — rollout-gated)

Named PMC, client, engineer, contractor and consultant users now use their invited email plus a stable password for routine sign-in. The first setup, and any later reset, requires one email OTP; verifying that OTP does **not** create an application session. Password completion establishes the credential, increments the user's credential version and invalidates older sessions. Unknown, inactive and removed-only identities receive the same public response but no usable challenge. Worker access remains QR/device based.

Project team managers can see only `Password not set` or `Password active`. Organisation owners/admins see the same state in the organisation roster and may correct a mistyped invitation email only before password enrollment; the correction revokes outstanding setup challenges and creates an attributable security audit event. Organisation role/add/remove authority remains owner-only.

Internal rollout order:

1. Rotate the exposed Coolify root token and `SMTP_PASS`; verify credential emails are delivered by the live SMTP provider.
2. Copy the database backup out of container `/tmp`, store it durably, and confirm daily automated backups plus a restore check.
3. Deploy the API first and confirm `20261012000000_internal_password_enrollment` applies cleanly; then deploy the web app.
4. Pilot setup, login, reset and old-session rejection with one controlled internal account.
5. Confirm invitation emails in the Team screen, correct any pre-enrollment typo, then enroll the remaining named users.

Existing construction/project data and existing password hashes are preserved by the additive migration. Do not run the destructive demo seed against production.

## Known product gaps (carried forward, not bugs)

- ~~**Checklist ↔ Review are decoupled**~~ **Resolved.** A submitted checklist now feeds the **PMC review queue**: the snapshot exposes `reviews[]` (every submitted-but-undecided inspection — a submitted checklist, the seeded review, and auto-created closing inspections), the review screen renders a queue selector, and each checklist `state` (pass/fail/na) maps to a review `result` (PASS/FAIL). The demo mirrors this locally (submit as engineer → it appears in the PMC's queue).
- **Swatches are CSS-placeholder gradients** standing in for material samples in the decision options. Real *site* photos are now first-class (progress-photo upload/gallery + geo/time stamp + tap-to-zoom, `7c-media`); the decision-option swatches remain as material-sample stand-ins.
- **Persona switcher** ("Viewing as") is now a dev-only device: gated behind `ALLOW_DEV_AUTH`/`VITE_ALLOW_DEV_AUTH` and hidden in the locked-down build, replaced by real sign-in + a full-screen auth gate (`7c-auth lockdown`, `docs/AUTH_LOCKDOWN.md`).
- **Display ids (`DL-…`, `ACT-…`, `INSP-…`) are global database primary keys** (DATA-01). Interim fix in place: the id sequence scans the whole table, so two projects never collide — but numbering is shared across projects (project B's first decision may be `DL-016`, not `DL-001`), and two *concurrent* creates can still race to the same next id and surface a `P2002`/500 (rare; retry succeeds). The durable fix is a schema migration: internal CUID primary key + a per-project display `code` with `@@unique([projectId, code])` allocated in a transaction, remapping every foreign key (`DecisionOption`, `Activity.decisionId`, `SiteMaterial.decisionId`, `Drawing.decisionId/activityId`, `InspectionItem`, media links) and every API lookup. Do it against a staging copy first — it rewrites live primary keys.

## Security follow-ups (tracked from the third-pass audit — not yet fixed)

These need larger, coordinated changes than a point fix; they are called out so they aren't lost. The immediately-exploitable findings from the same audit (OTP code exposure, worker-token PII reads, dev-auth in prod, demo-password provisioning, push SSRF, inspection state-machine, upload MIME/size) are **fixed**.

- ~~**Private, authorized file delivery (media & drawings).**~~ **Resolved.** `GET /media/:id` and `GET /drawings/rev/:id` now require a short-lived `?t=` token (`SignedUrlService`, HMAC over `kind:id:exp`, default 60 min via `FILE_URL_TTL_SEC`) minted only while building the RBAC-filtered snapshot / on upload — a bad/missing token is `403`. Public bucket URLs are never persisted or emitted; in S3/R2 mode the serve endpoint presigns a short-lived GET on demand, so the **bucket must be private** (no public-read). `Cache-Control` dropped from a year to `private, max-age=<ttl>`. A leaked file URL now expires quickly and a revoked user can't mint fresh tokens. Frontend resolves the signed API-relative paths against the API base (`resolveMediaUrl`/`resolveDrawingUrl`).
- **Authenticated realtime (socket.io).** `handleJoin` lets any socket join any `project:<id>` room with no token check; the emitted payload is only a `{ projectId }` "refetch" signal (no data), so the exposure is limited to change-timing for a known project id. Fix: authenticate the socket handshake with the JWT and authorize the room join against the token's project. Low data-exposure, deferred behind the file-delivery work.
- **Outbox replay scope-generation (WEB-02 deepening).** The offline queue is keyed by user+project, but a flush that begins before an identity/project switch can still resolve against the new scope, and the one-time legacy-queue migration is claimed by whoever hydrates first. Fix: stamp a scope generation on each op and abort/ignore a flush whose generation no longer matches the active session. Client-only, shared-browser edge.
- **CORS & JWT are fail-soft by design.** With `CORS_ORIGINS` unset in prod the API reflects the request origin (safe for a bearer-token, cookieless API, but permissive); with `JWT_SECRET` unset it derives a stable secret from `DATABASE_URL`. Both degrade safely rather than failing closed. Set `CORS_ORIGINS` and `JWT_SECRET` explicitly in prod (see GO_LIVE). A future hardening could make these fail-closed behind an explicit opt-out.
- **No per-request token revocation.** A JWT stays valid up to its 12h expiry after a role change or removal; access-resolution changes only affect *newly issued* tokens. Acceptable for now (short expiry, no refresh tokens); a deny-list or short-lived-access-token + refresh design is the durable answer.
- **Presigned drawing uploads have no server-enforced size cap.** The `/drawings/presign` PUT URL lets a PMC upload directly to the bucket; the API caps the *reported* `sizeBytes` on issue but the S3 PutObject presign itself carries no `content-length-range` condition, so a PMC could PUT an oversized object. Low risk (PMC-only, trusted), but the durable fix is a POST-policy upload with a size condition + a server-created upload-session record the `issue` call must match (rather than trusting the echoed `storageKey`; the key is now at least bound to the caller's project prefix).
